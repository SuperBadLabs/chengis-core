# chengis-core v0.2 — Executor Backends

The chengis-core release that closes the executor gap exposed by anvil's
wild-corpus matrix. v0.2 ships the protocol layer beneath `StepDispatcher`
that lets steps execute in pluggable backends (local-shell, Docker,
Kubernetes) instead of always running in-process via
`chengis.engine.process/execute-command`.

This board is the chengis-core slice of
[Operation Brasstacks](https://github.com/SuperBadLabs/anvil/blob/master/docs/brasstacks/board.md)
Phase 1. See the master board for the cross-product strategic context;
this document tracks the chengis-core implementation tranches.

## Why this exists

anvil v0.3 ships a single-host shell executor. Real-world Jenkinsfiles
ship `agent { docker }`, `agent { kubernetes }`, `tool('jdk_17_latest')`,
and `withCredentials([...])` — none of which work in a single-host
shell. anvil silently skips agent bodies, returns empty strings for
tools, and binds credentials to `""`. The wild-corpus matrix
(15 real-world OSS Jenkinsfiles) parses 15/15 but builds 0/15 to real
artifacts.

chengis-core v0.2 closes this honestly:

1. A new `ExecutionBackend` protocol — pluggable, with LocalShell as the
   reference impl, Docker as the unblocking impl, Kubernetes as the
   v0.3 follow-on
2. Honest build-result classification (NEUTRAL / UNSUPPORTED added to
   the result enum)
3. Tool installer registry with per-version cache (JDK/Maven/Gradle/Node/etc.)
4. Credentials binding pipeline that injects real values into env +
   config files (`~/.m2/settings.xml`, `~/.npmrc`)
5. Plugin-step framework + top-N first-class implementations
6. 7-day soak under 100 concurrent Docker builds

After v0.2, anvil v0.4 layers Jenkinsfile-name mappings over the new
engine. The wild-corpus matrix is expected to score 13/15 real builds
(the 2 holdouts are `agent { kubernetes }` projects, which Phase 3's
chengis-core v0.3 EX6 addresses).

## Naming note

The existing `chengis.agent.*` namespaces in this repo (`core`,
`worker`, `client`, `heartbeat`, `artifact-uploader`) implement the
**distributed-agent daemon** — a separate process that receives build
dispatches from a master controller over HTTP. That's distinct from the
**execution backend** abstraction in CC2-EX1a, which is the
in-process pluggable layer that decides whether a step runs in the
local shell, a Docker container, or a K8s pod.

To avoid collision, the new protocol lives at `chengis.engine.backend`
(not `chengis.engine.agent`). The two layers compose: a distributed
agent worker can hold its own ExecutionBackend; a single-host anvil
install uses the backend directly without an agent daemon.
